# BiliTune 参考项目功能汇总

> 基于两个成熟 B 站音乐客户端的源码深度分析：**biu** (Electron 桌面端) 和 **BBPlayer** (React Native 移动端)

---

| 维度 | biu | BBPlayer |
|------|-----|----------|
| 定位 | B站音乐桌面播放器 | B站音乐移动播放器 |
| 技术栈 | Electron 38 + React 19 + Rsbuild | React Native + Expo SDK 56 |

## 二、核心播放系统

### 2.1 双类型播放

两个项目都支持从 B 站的两种内容源播放音频：

| 类型 | 说明 | API 来源 |
|------|------|---------|
| `mv` (视频稿件) | 普通视频的音频流 | DASH 流 (`/x/player/playurl`) |
| `audio` (音频区) | B站音乐区专用音频 | 音频流 (`/audio/music-service-c/url`) |

**biu 实现**：`play-list.ts` 根据 `type` 字段选择不同 API，MV 走 `getDashUrl`，Audio 走 `getAudioUrl`

**BBPlayer 实现**：`lib/api/bilibili/api.ts` 的 `getAudioStream` 统一处理，支持 DASH/DURL/杜比/Hi-Res/响度归一化

### 2.2 DASH 流解析

B站视频使用 DASH 格式返回多音轨：

```
响应数据结构:
data.dash.audio[]     → 音频轨列表 (64kbps ~ 320kbps ~ Hi-Res)
data.dash.dolby.audio[] → 杜比音轨
data.dash.flac        → 无损 FLAC 音轨
```

| 音质等级 | 描述 | 带宽 |
|---------|------|------|
| 30216 | 64kbps | 低音质 |
| 30232 | 128kbps | 标准 |
| 30280 | 320kbps | 高品质 |
| 30250 | FLAC | 无损 |
| 30251 | Dolby Atmos | 杜比全景声 |

### 2.3 播放模式

| 模式 | 说明 |
|------|------|
| 顺序播放 | 按列表顺序，播完停止 |
| 列表循环 | 播完最后一首回到第一首 |
| 单曲循环 | 重复当前曲目 |
| 随机播放 | 随机选择下一首 |

### 2.4 播放速率

支持 0.5x ~ 2.0x 倍速调节。

### 2.5 音频 URL 自动刷新

B站音频 URL 含 `deadline` 参数（通常 6 小时有效），过期后需重新获取：

- **biu**：`refreshCurrentAudioSource()` 检测 deadline，过期自动重新调用 API
- **BBPlayer**：Orpheus 原生层通过 `orpheus://` 协议自动处理

### 2.6 响度均衡 (Volume Normalization)

**BBPlayer 独有功能**：B站 API 返回 `measured_i` / `target_i` 值，用于计算每首歌的增益补偿，使不同音量的歌曲播放时音量一致。

### 2.7 播放记录上报

仅对视频类型（非本地文件）上报心跳：

1. **播放开始** → 调用 `heartbeat/start` 上报
2. **播放中** → 每 15 秒心跳一次 `heartbeat/web`
3. **播放结束** → 最终上报

## 三、下载系统

### 3.1 biu 下载引擎 (桌面端)

完整的分块并发下载器：

```
流程：获取 URL → Content-Length → 按 10MB 分块 → got + p-queue (并发5) 下载 → 合并 → FFmpeg 转换
```

| 特性 | 说明 |
|------|------|
| 分块大小 | 10MB |
| 并发线程 | 5 |
| 断点续传 | ✅ 检测已存在分块大小 |
| 队列并发 | 最多 3 个同时下载任务 |
| 暂停/恢复 | ✅ |
| 重试 | ✅ 失败自动重试 |

**FFmpeg 转换**：
- 纯音频：`ffmpeg -i audio -c:a copy output.m4a` / `.flac`
- 音视频：`ffmpeg -i video -i audio -c:v copy -c:a copy -shortest output.mp4`

### 3.2 BBPlayer 下载 (移动端)

基于 Orpheus 原生模块：

| 特性 | 说明 |
|------|------|
| 同时下载数 | 可配置 (1/2/3/6) |
| 多任务下载 | ✅ `multiDownload` |
| 暂停/恢复 | ✅ `resumeDownload` |
| 重试 | ✅ `retryDownload` |
| 离线播放 | ✅ 通过 `orpheus://` 协议播放本地缓存 |

### 3.3 下载格式支持

| 格式 | 桌面端 (biu) | 移动端 (BBPlayer) |

| M4A/AAC | ✅ | ✅ |

## 四、登录系统

### 4.1 支持的登录方式

| 方式 | biu | BBPlayer |
|------|-----|----------|
| 二维码扫码 | ✅ | ✅ |
| 短信验证码 | ✅ | ✅ (含极验) |
| 密码登录 | ✅ | ❌ |
| Cookie 手动填入 | ❌ | ✅ |

### 4.2 二维码登录流程

```
1. 调用 /x/passport-login/web/qrcode/generate 获取二维码 URL + qrcode_key
2. 前端渲染二维码（用户用B站APP扫码）
3. 轮询 /x/passport-login/web/qrcode/poll?qrcode_key=xxx
   - 86101: 未扫码
   - 86102: 已扫码未确认
   - 0: 确认登录 → 从 Set-Cookie 提取 SESSDATA / bili_jct
```

### 4.3 短信登录流程

```
1. 获取极验参数 (geetest)
2. 用户完成极验验证 → 获取 challenge/validate/seccode
3. 调用 /x/passport-login/web/sms/send 发送验证码
4. 用户输入验证码
5. 调用 /x/passport-login/web/login/sms 完成登录
```

### 4.4 Cookie 自动刷新

```
1. 检查 cookie_info 判断是否需要刷新
2. RSA-OAEP 加密 "refresh_{timestamp}" → correspondPath
3. 访问 /correspond/1/{correspondPath} → 提取 refresh_csrf
4. 调用 /x/passport-login/web/cookie/refresh 刷新 Cookie
5. 调用 /x/passport-login/web/confirm/refresh 确认并更新 refresh_token
```

## 八、弹幕系统

### 8.1 弹幕获取

B站使用分段弹幕（protobuf 格式）：

```
API: /x/v2/dm/web/seg.so?type=1&oid={cid}&pid={aid}&segment_index={6位时间}
```

| 参数 | 说明 |
|------|------|
| oid | cid |
| pid | aid |
| segment_index | 6分钟一段，第N段 = N |

### 8.2 弹幕解析

protobuf 解码为弹幕对象数组，每条弹幕包含：
- 时间点、内容、颜色、字号、模式（滚动/顶部/底部）

## 十五、极验验证码处理

两个项目都实现了极验验证码的自动处理：

```
1. API 响应包含 v_voucher
2. 调用 /x/gaia/vgate/register 获取极验参数
3. 加载极验 JS 脚本，弹出验证
4. 用户完成验证 → 获取 challenge/validate/seccode
5. 调用 /x/gaia/vgate/validate 验证
6. 将 gaia_vtoken 附加到原请求重新发送
```
